## Overview

Measure captures your app's network requests, responses, and failures. Each request records its URL, HTTP method, status code, and how long it took. Similar URLs are grouped into [endpoint patterns](/docs/network-monitoring/endpoint-patterns), so the metrics aggregate per endpoint and you can see how each API performs in production.

Request and response headers and bodies are collected only for the URLs you opt into, so sensitive data stays out by default. Choose those URLs from the dashboard through the [HTTP events](/docs/configuration-options#http-events) settings.

Beyond HTTP, Measure records [connectivity changes](/docs/network-monitoring/connectivity-changes) so you can tell when the network itself, not your API, was the problem.

## Tracked automatically

Requests from the clients below are captured with no per-request code. On Android, add the [Measure Android Gradle Plugin](/docs/getting-started/android#3-add-the-gradle-plugin).

### Android

Requests from **OkHttp** (versions `4.7.0` to `5.3.2`) and **HttpURLConnection** (SDK `0.18.0` and later) are tracked, along with any library built on OkHttp. OkHttp requests outside that version range aren't tracked.

### iOS

Requests from **URLSession** are tracked, including any made by a third-party library built on it.

### Kotlin Multiplatform

Requests from **Ktor** are tracked when it runs on the OkHttp (Android) or Darwin (iOS) engine. Tracking happens in the native layers, so the [Kotlin Multiplatform guide](/docs/getting-started/kotlin-multiplatform) covers the setup. A different engine like CIO isn't tracked, so [track those manually](#track-a-request-manually).

## Add an interceptor

A few clients need a one-time interceptor or dependency change before Measure can see their requests.

### Dio (Flutter)

Add the `measure_dio` package and register `MsrInterceptor` on your `Dio` instance:

```yaml
dependencies:
  measure_dio: latest-version
```

```dart
final dio = Dio();
dio.interceptors.add(MsrInterceptor());
```

For Flutter's `http` package or any other client, [track requests manually](#track-a-request-manually).

### Retrofit (Android)

Retrofit runs on OkHttp, but pulls it in as a transitive dependency, and auto-instrumentation doesn't reach transitive dependencies. If your project depends on Retrofit alone, use one of these.

**Declare OkHttp as a direct dependency.** Pin an OkHttp version in your `build.gradle.kts`, and auto-instrumentation works with no code changes:

```kotlin
dependencies {
    implementation("com.squareup.retrofit2:retrofit:<version>")
    implementation("com.squareup.okhttp3:okhttp:<version>")
}
```

**Add the interceptor manually.** Build the `OkHttpClient` you pass to Retrofit with Measure's interceptor and event listener factory:

```kotlin
val client = OkHttpClient.Builder()
    .addInterceptor(MeasureOkHttpApplicationInterceptor())
    .eventListenerFactory(MeasureEventListenerFactory(null))
    .build()

val retrofit = Retrofit.Builder()
    .baseUrl("https://api.example.com")
    .client(client)
    .build()
```

### React Native

On React Native, `fetch` and `XHR` run on OkHttp on Android and URLSession on iOS. iOS tracks requests automatically, Android needs setup, and capturing response bodies needs an interceptor.

#### Android

Configure a custom `OkHttpClient` with `OkHttpClientProvider` in `MainApplication.kt`. Without this, HTTP events aren't tracked on Android:

```kotlin
OkHttpClientProvider.setOkHttpClientFactory(object : OkHttpClientFactory {
  override fun createNewNetworkModuleClient(): OkHttpClient {
    return OkHttpClient.Builder()
      .cookieJar(ReactCookieJarContainer())
      .addInterceptor(MeasureOkHttpApplicationInterceptor())
      .eventListenerFactory(MeasureEventListenerFactory(null))
      .build()
  }
})
```

#### iOS

iOS tracks requests automatically. To capture responses too, add `MSRNetworkInterceptor` to the session configuration in `AppDelegate.mm`:

```objc
RCTSetCustomNSURLSessionConfigurationProvider(^NSURLSessionConfiguration *{
  NSURLSessionConfiguration *configuration = [NSURLSessionConfiguration defaultSessionConfiguration];
  [MSRNetworkInterceptor enableOn:configuration];
  return configuration;
});
```

Response bodies are recorded only for the URLs you opt into, set with the **Track HTTP response body for URLs** option in [Configuration options](/docs/configuration-options#http-events).

For clients outside `fetch` and `XHR`, [track requests manually](#track-a-request-manually).

## Track a request manually

For any HTTP client Measure doesn't instrument automatically, record each request yourself with `trackHttpEvent`. Bracket the request with `getCurrentTime` for the start and end time, so the duration stays right even if the device clock shifts. See [Track HTTP events](/docs/api-reference#track-http-events) for the full signature and the optional fields you can add.

<Tabs groupId="platform" items={["Android", "iOS", "Flutter", "React Native", "Kotlin Multiplatform"]} persist>
<Tab value="Android">

```kotlin
import sh.measure.android.Measure

val startTime = Measure.getCurrentTime()
// make the request
val endTime = Measure.getCurrentTime()

Measure.trackHttpEvent(
    url = "https://api.example.com/users",
    method = "GET",
    startTime = startTime,
    endTime = endTime,
    statusCode = 200,
)
```

## Configuration options

Control what network data Measure collects, and for which URLs, from the dashboard through [Adaptive Capture](/docs/configuration-options), so you can change these without shipping a new build:

- **Sampling rate**: how many HTTP events are collected. All of them by default, down to a floor of 0.001%.
- **Collect or ignore URLs**: turn collection on or off for specific URLs, matched exactly or with `*` wildcards.
- **Request body collection**: the URLs whose request bodies and headers are recorded. Off by default.
- **Response body collection**: the URLs whose response bodies and headers are recorded. Off by default.
- **Blocked headers**: headers to drop by name. Sensitive ones like `Authorization` and `Cookie` are never collected, whether or not you list them.

See [HTTP events](/docs/configuration-options#http-events) for each setting.
